# Как написать эффективный CLAUDE.md

## Золотые правила

### 1. Держите его менее 50 строк

Каждый токен на счету. Раздутый CLAUDE.md = меньше места для реальной работы.

### 2. Правила, а не документация

CLAUDE.md — это **справочник**, а не руководство.

```markdown
# ПЛОХО
## Система аутентификации
Наша аутентификация использует JWT-токены, хранящиеся в HTTP-only cookies.
Процесс работает следующим образом: сначала пользователь отправляет учетные данные...
[еще 200 строк]

# ХОРОШО
## Auth
- JWT в HTTP-only cookies
- Подробнее: docs/auth.md
- Точка входа: server/auth/
```

### 3. Документируйте то, в чем Claude ошибается

Добавляйте правила реактивно, основываясь на ошибках:

```markdown
# ПЛОХО: Предварительное документирование всего
- Используйте табы вместо пробелов
- Имена переменных должны быть описательными

# ХОРОШО: Правила из реальных ошибок
- НИКОГДА не используйте inline-стили — используйте CSS-классы из assets/css/
- API-маршруты ДОЛЖНЫ валидироваться через Zod перед обработкой
```

### 4. Объясняйте почему, предлагайте альтернативу

```markdown
# ПЛОХО
- Никогда не используйте any

# ХОРОШО
- Никогда не используйте `any` — используйте `unknown` и сужайте тип, или определите правильный тип
  Почему: `any` ломает типобезопасность и скрывает баги
```

### 5. Ссылайтесь, а не копируйте

Указывайте на документацию, не копируйте её:

```markdown
# ПЛОХО: Полная документация API в CLAUDE.md

# ХОРОШО
## Разработка API
- Читать: docs/api-patterns.md при создании эндпоинтов
- Читать: docs/database.md при изменении схемы
```

## Рекомендуемая структура

```markdown
# {Название проекта}

## Видение
{Одно предложение, описывающее цель проекта}

## Стек
- Frontend: {фреймворк} | Точка входа: {путь}
- Backend: {фреймворк} | Точка входа: {путь}
- База данных: {тип}

## Критические правила
- {Правило 1 с обоснованием}
- {Правило 2 с обоснованием}
- Git: Спрашивайте перед commit/push

## Паттерны
{Пример обязательного паттерна}

## Команды
{команда для разработки}
{команда для тестов}
{команда для сборки}

## Контекст
Читать: .claude-project/context.md
```

## Что НЕ включать

- Общие советы ("пишите чистый код")
- Стандартное поведение Claude ("обрабатывайте ошибки правильно")
- Всю документацию целиком
- Противоречивые правила
- Более 50 строк

## Что включать

- Специфичные для проекта ограничения
- Критические паттерны, отличающиеся от стандартных
- Точки входа и ключевые файлы
- Команды для разработки/тестов/сборки
- Ссылку на файл контекста

## Поддержание

1. **Добавляйте правила, когда Claude делает ошибки**
2. **Удаляйте правила, которые больше не актуальны**
3. **Проверяйте ежемесячно** для очистки
